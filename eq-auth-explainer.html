1

Worker claims their wallet

Tradesperson received a link. They're verifying it's them before seeing their licences and certs.

👷 Worker
📱 cards.eq.solutions
🔐 GoTrue / Supabase
  • Opens the join link
    cards.eq.solutions/join?token=abc123
  • Types their mobile number
    Cards sends an SMS OTP via GoTrue → Twilio
  • Receives SMS, types the 6-digit code
    Cards calls verifyPhoneOtpOnly() → gets a GoTrue access token
  • Cards calls shell-join-tenant
    Passes phone + access token + join token → Shell checks shell_control.users, creates membership
  • ✅ Wallet screen loads
    Worker sees their licences and certificates. Cards session stays in-app.
What's stored: A GoTrue session lives in supabase-js local storage (Cards only). No shell session cookie is set — this user doesn't have Shell access yet.
2

Admin provisions a new workspace ✦ Just built

Brand-new company. Platform admin sent a /provision link. Nobody exists yet.

🧑‍💼 Admin
📱 Cards
🏠 Shell functions
🏠 Shell app
  • Opens the provision link
    cards.eq.solutions/provision?token=xyz
  • Types mobile number → receives SMS → types code
    Same phone OTP flow as Scenario 1. Cards gets a GoTrue access token.
  • Cards calls shell-provision-tenant
    Sends phone + access token + provision token → Shell validates JWT, creates tenant + user + membership, consumes provision token (one-time only)
  • Shell returns tenant_slug + access_token
    Cards captures both values, then signs out of its own GoTrue session
  • Browser opens Shell with a hidden token
    core.eq.solutions/acme#sh=<token> — the # means it never reaches any server
  • Shell detects the fragment, strips it immediately
    URL is cleaned via history.replaceState before anything else runs
  • Shell calls shell-handoff-provision (same-origin)
    Same-origin POST with the token → Shell validates the JWT, looks up the user, mints a 7-day session cookie
  • ✅ Admin lands in /acme — already logged in
    No second OTP. Shell session cookie is set. Cards returned to its sign-in screen in the background.
Why the #sh= trick? Setting a cookie across two different domains is blocked by browsers. Instead, Shell's own JavaScript calls its own Netlify function with credentials: 'include' — same-origin request, so the browser allows the Set-Cookie. The token travels only inside the browser, never to a server as a URL parameter.

Quick decision tree

Which scenario applies?

Is there already a valid session cookie?
YES → verify-shell-session passes → straight in ✅
NO → Where is the user?
→ /provision?token= → Scenario 2 (new org admin)
→ /join?token= → Scenario 1 (worker claim) — or Gate A if not pre-registered
→ cards.eq.solutions with email → Scenario 4 (email OTP)
→ core.eq.solutions → Scenario 3 (Shell phone login)
→ Returning after 7+ days → Scenario 6 → then Scenario 3
