⚑ Critical Findings & Advice — from enterprise standards review · updated 2026-06-13
Security
No secret tier classification
Every var is tagged Required/Optional but nothing distinguishes a public PostHog key from SUPABASE_SERVICE_ROLE_KEY. A deploy log leaking the anon key is a nuisance. The same leak for the service role key is a breach.
Action: Vars are now auto-tagged T1/T2/T3 in this catalog. T3 vars must never appear in logs, error responses, Sentry breadcrumbs, or Next.js error overlays. Add each T3 name to your Sentry beforeSend scrub list.
Security
Netlify preview deploys inherit T3 secrets
By default Netlify injects env vars into all contexts — production, deploy previews, and branch deploys. Any contributor who can open a PR can trigger a preview and exfiltrate SUPABASE_SERVICE_ROLE_KEY or EQ_SECRET_SALT via a Netlify Function.
Action: In Netlify → Site Settings → Environment Variables, scope all T3 vars to Production context only. Analytics (PostHog, Clarity, Sentry DSN) can stay on All Contexts. Resend key → production only (or use test-mode key for previews).
Security
EQ_SECRET_SALT shared across 4 EQ sites amplifies blast radius
A single HMAC salt across Shell, Field, Service, and Cards means compromising any one site's Netlify env compromises HMAC integrity on all others simultaneously. sks-nsw-labour also holds the same salt but is decommissioning — blast radius shrinks to 4 sites at cutover.
Architecture fix: Derive per-app salts via HKDF from a root key — each app gets a unique salt, limiting compromise to one site. Until then: treat a salt rotation as a fleet-wide incident requiring simultaneous update across all 4 EQ Netlify sites (+ sks-nsw-labour until cutover). See Open Actions → Rotation runbook.
Security
EQ_SERVICE_JWT_SECRET fallback collapses trust boundary
If unset, EQ Service silently uses SUPABASE_JWT_SECRET for application-level JWTs. Tokens meant to be app-scoped can now be validated against Supabase's auth system and vice versa — if the app JWT key is compromised, an attacker can forge Supabase session tokens.
Action: Remove this fallback before production. Set EQ_SERVICE_JWT_SECRET explicitly in all environments. Never share a signing key between two systems with different trust levels.
Security
STAFF_CODE / MANAGER_CODE are security theatre
These are low-entropy shared PINs (e.g. 2026, SKSNSW) known to every staff member. They are cataloged in plain text here. Storing them alongside SUPABASE_SERVICE_ROLE_KEY creates false equivalence and a misleading security posture.
Action: Do not conflate with cryptographic secrets. Rotate after every staff departure. Migration path: per-user passkey or OTP before any public-facing deployment.

Operations
No rotation policy or runbook
No secret has a documented rotation cadence, rotation trigger, or step-by-step procedure. For EQ_SECRET_SALT, a rotation requires coordinating 5 Netlify sites — without a runbook, this will be executed incorrectly under pressure.
Cadence targets: API keys (Resend, Anthropic) → 90 days. Infra secrets (Supabase service role) → 180 days. All T3 → immediate on suspected compromise. Runbook: generate → update Netlify (production context) → redeploy → verify health → revoke old at source.
Operations
CRON_SECRET Vault sync has no verification step
CRON_SECRET must match the Supabase Vault value. A rotation of one without the other silently stops all scheduled jobs with no alert. There is currently no dead-man's-switch monitoring for cron job absence.
Action: After rotation: (1) update Netlify, (2) update Vault via vault.update_secret(), (3) trigger cron manually + verify 200, (4) set PostHog alert for cron job absence within expected window.
Operations
BATTLE_TEST_* vars are in production context
Test fixture UUIDs and emails are set as production env vars on EQ Service. If a code path accidentally reads them in production, it affects real data. They also clutter the production env with non-production config.
Action: Scope to Branch deploys only in Netlify. If battle tests must run against production data, use a dedicated test tenant UUID in the database — not an env var.
Completeness
No format / validation requirements documented
Secrets with strict format requirements (CRON_SECRET = 32-byte hex, SITE_CREDENTIALS_KEY = ≥32 chars base64) have no generation commands documented. A wrong format produces a silent failure or runtime panic — not a helpful error.
Generate 32-byte hex: node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
Generate base64: node -e "console.log(require('crypto').randomBytes(48).toString('base64'))"
