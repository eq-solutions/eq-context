  - **No admin-edit feature.** Editing a worker's record erodes the worker-ownership/portability moat (the thing that separates Cards from an employer HR DB / competes with MyPass). Flag-don't-edit keeps ownership with the worker AND the fix travels with them to the next employer.
- The one theoretical gap I raised — decline is an *intake-gate* verb, so an already-*connected* worker with bad data would want a soft "flag for fix" nudge rather than a revoke — Royce judged **hair-splitting; left alone.** Not building it.

## Notes
- Framework used to get there (for future reference): admin actions split into **correcting** (record→reality, fine) / **overriding** (admin substitutes judgment, dangerous) / **owning** (admin becomes the authority, breaks the product). Rails that make any admin write safe: scoped to connected workers, attributed + reasoned into `audit_log`, visible to the worker, reversible by the worker. Tier by data type: profile fields = fine; credentials = need rails; consent/sharing = hard no.
- Reality check that reframed it: the platform already does admin-fixing at onboarding (`eq_cards_admin_upsert_worker` pre-populates profiles + creds) — so "admin fixes things" isn't a new philosophy, and "no admin edit" just means corrections leak to dev-run SQL.
