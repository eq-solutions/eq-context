## Session — eq-field — ✅ SKS leave-shows-0 FIXED & VERIFIED (v3.5.291, prod-clean v3.5.292)

**Multi-session "SKS leave shows 0" bug — RESOLVED and verified live on `core.eq.solutions/sks/field?tab=leave`** (PENDING 1 / OFF THIS WEEK 10 / APPROVED 15; sidebar badge 1; "Out: Simon, Todd, Luke").

### True root cause (proven end-to-end, not inferred)
The prior "canon:false / TENANT.ORG_SLUG wrong" hypothesis was a **red herring**. Routing was correct all along — an on-screen BOOT_DIAG banner read off a screenshot of the embedded iframe showed `slug=sks, window.SB_URL=ehow, hasLeaveAdapter=true, canon=TRUE`. The leave **READ never ran**:
1. `leave.js` is lazy-loaded → `initApp()`'s boot `loadLeaveRequests()` is skipped (`loadLeaveRequests` undefined at that point);
2. `renderLeave()` only rendered from the in-memory list — never loaded;
3. realtime merges CHANGES but does no initial read;
4. the 30s poll (`refreshData→loadLeaveRequests`) is suppressed while realtime is connected.
On a deep-linked `?tab=leave` (exactly how Core embeds Field), `leaveRequests` was never populated. The panel "↺ Refresh" was `onclick=renderLeave()` — a pure re-render, so a manual refresh never loaded it either. The 31 rows were never lost — never fetched.

### Shipped
- **v3.5.286 (#439)** — canonical gates (leave/roster/timesheets) keyed on resolved DB (`window.SB_URL`=ehow) as well as slug; `window.SB_URL` exposed. Necessary (canon was slug-fragile), not sufficient.
- **v3.5.287 (#440)** — `sbFetch` refreshes `window.SB_URL` every call.
- **v3.5.288–290 (#442/#444/#445)** — temporary diagnostics (Sentry LEAVE_DIAG, on-screen BOOT_DIAG/LEAVE_DIAG banners, `__eqDiag` trace).
- **v3.5.291 (#446)** — **THE FIX:** `renderLeave()` → `_ensureLeaveLoaded()` (cached-promise one-shot fetch on first tab open + re-render).
- **v3.5.292 (#447)** — removed all temporary diagnostics (were user-visible). All live on field.eq.solutions, verified serving.

### Diagnostic lesson (load-bearing for future Shell-embedded Field debugging)
The embedded Field iframe under `core.eq.solutions` is **cross-origin AND storage-partitioned**: can't script it, its localStorage is an isolated partition, and it emits NOTHING to Sentry (verified: pipe works first-party, silent embedded). The ONLY observation channel that worked was **painting the runtime state into a fixed on-screen banner and reading it off a screenshot** (screenshots capture rendered iframe pixels). Reach for that early next time instead of console round-trips.
