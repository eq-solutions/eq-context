## Notes
- **PREMISE CORRECTION, load-bearing:** the canonical adapters are NOT default-off. `TIMESHEETS_/ROSTER_/LEAVE_CANONICAL_TENANTS` all contain `'sks'`, plus a DB-ref fallback matching ehow. **SKS — the only live tenant — runs the canonical normalized path.** The "DEFAULT OFF" comments in the adapter files describe the original state, not the current one. Three separate task briefs this session asserted "default OFF"; all three were wrong. Banked to memory.
- **`_loadFullDataForExport` was already broken on SKS today**, at 80 staff — `sbFetchAll`'s default `order=id` 400s against `app_data.timesheets`, which keys on `timesheet_id` and has no `id`. Not a 1500-scale hypothetical.
- **Why naive paging would have been worse than the bug:** timesheets/roster `toWideList` GROUPS (many per-day rows → one week row) and runs PER RESPONSE, so concatenating pages emits a duplicate partial wide row for every person/week straddling a boundary.
