## Notes
- **The whole premise inverted on inspection, and the inverting evidence was `public.audit_log`.** The 6 rows read exactly like pending phone-only onboarding stubs — `email`, `phone` AND `name` all NULL, `pin_hash` NULL, role `employee`, `last_login_at` NULL, tenants SKS + Personal Wallet. Creating licences and inviting members needs a working Supabase JWT, so an `auth.users` row existed and was removed later.
- **TRAP, banked to memory: `last_login_at IS NULL` is not evidence of "never signed in"** — it's null on 60 of 72 `shell_control.users` rows; the column isn't maintained on the Cards/OTP path. It is the single field that made these look untouched, and it's what the task brief rested on.
- Corroboration, all live-verified on jvkn: `public.profiles` for all six **gone** (cascaded), `workers`/`licences`/`org_memberships` 0. `eq_cards_delete_account` checked via `pg_get_functiondef` **only soft-deletes** — never touches `auth.users` — so the removal was **manual admin cleanup**, not the app's delete path. One id (`a248470c`) was created 88 seconds after another (`2a467a34`) was deleted — same person re-signing-up, i.e. self-signup churn someone tidied by hand.
- **`backfill-auth-users.ts` was the wrong remedy twice over.** It calls `createUser({ id, email, email_confirm: true })` — `email` is NULL on all six, so GoTrue 422s and they'd land in `failures`. And if it HAD worked it would have re-minted login identities for people who exercised deletion. It also loops all 72 `shell_control.users`, not just the orphans. It has a phone/email collision guard but no deleted-account guard.
- **Root cause is structural, not historical:** `eq_finalize_deleted_accounts` runs `SET search_path = public` so it cannot see `shell_control`, and `shell_control.users` has no FK to `auth.users` so nothing cascades. Every finalised account leaves one behind. 1 profile is already pending erasure, so this was live, not theoretical.
- **Design rule in the fix: provenance, never a predicate.** Deliberately NOT `WHERE NOT EXISTS (SELECT 1 FROM auth.users …)`. The Shell mints its own session JWT and a Shell-only account signs in fine with no `auth.users` row (same fact recorded in the Emma Curth merge runbook, 2026-07-14), so a missing one is NOT evidence of deletion — a blanket sweep could delete live accounts. Step 3 is keyed off `authDeleted`, not `ready_user_ids`: a failed GoTrue delete must leave the Shell row so the next run retries the pair together.
- **FK shape on `shell_control.users`, verified live** — `user_tenant_memberships`, `user_security_groups`, `pin_reset_tokens.user_id`, `revoked_sessions.user_id` CASCADE. **Five referrers are ON DELETE NO ACTION and will abort a delete**: `user_invites.invited_by`, `cards_field_approvals.approved_by_user_id`, `pin_reset_tokens.created_by`, `phone_link_review.user_id`, `tenant_role_overrides.updated_by`. All five are 0 for these six. Hence per-id deletes, not one batch — one blocked account would roll back the lot; a `shell_error` is 207, not fatal, since the auth identity is already gone.
- `public.audit_log` preserved throughout — those 52 rows are the only remaining evidence these accounts existed, and `audit_log.user_id` has no FK to `shell_control.users`.
