## Session 5 — EQ Service Shell SSO fix (this session)

### Problem
`core.eq.solutions/sks/service` showed Service's own login page instead of auto-authenticating via Shell. Quotes, Cards, Field all worked. Service SSO had never successfully fired.

### Root cause chain (3 bugs, all fixed)

**Bug 1 — Edge runtime crypto failure (proxy.ts)**
Shell SSO exchange was in `proxy.ts` (Next.js 16 middleware, runs on Netlify edge/Deno). `node:crypto` (`createHmac`, `timingSafeEqual`, `Buffer`) silently fail on the Deno-based edge runtime. The `try-catch` in `verifyShellCookie` caught the error and returned `null` → SSO skipped → login page.

**Fix:** Moved all HMAC verification, `generateLink`, and `verifyOtp` logic to a new Node.js API route `/app/api/shell-sso/route.ts`. `proxy.ts` now only detects `eq_shell_session` cookie and redirects to `/api/shell-sso?next=<path>`.

**Bug 2 — Wrong redirect hostname**
`/api/shell-sso` built its success redirect by cloning `request.nextUrl`. In Netlify's server handler context, `nextUrl.host` is an internal handler hostname — the redirect landed on `core.eq.solutions/dashboard` (Shell's assets loaded instead of Service's). Service loaded blank.

**Fix:** Replaced `request.nextUrl.clone()` with `new URL(safePath, process.env.NEXT_PUBLIC_SITE_URL)`.

**Bug 3 — HMAC key mismatch (EQ_SESSION_SALT vs EQ_SECRET_SALT)**
Shell signs `eq_shell_session` with `EQ_SESSION_SALT` (per-consumer key isolation, S2-9). Service's `verifyShellCookie` was reading `process.env.EQ_SECRET_SALT` only. Different keys → HMAC never matched → every SSO attempt silently fell through to `/auth/signin`.

**Fix:** `const salt = process.env.EQ_SESSION_SALT ?? process.env.EQ_SECRET_SALT` — mirrors Shell's resolution order.

### Also confirmed
- `VITE_SERVICE_URL=https://service.eq.solutions` was already set on Shell Netlify (since 2026-05-28) — Shell was in COOKIE_AUTH mode
- `service.eq.solutions` has no `X-Frame-Options` headers — iframe loading was never blocked
- `EQ_SESSION_SALT` values match on both Shell and Service sites
- Salts (`EQ_SECRET_SALT` + `EQ_SESSION_SALT`) confirmed identical between the two sites

### Bug 4 — Infinite redirect loop (PR #270, Session 6)
proxy.ts intercepted ALL non-bridged requests with an eq_shell_session cookie —
including /auth/signin. When shell-sso fails: fails → /auth/signin → proxy
intercepts → shell-sso → fails → /auth/signin → … → ERR_TOO_MANY_REDIRECTS.

**Fix (f3b1c5e):** `isSsoExempt = alreadyBridged || pathname === '/api/shell-sso' || isPublicPath(pathname)` — public paths skip the SSO redirect, so a failed shell-sso always lands at /auth/signin and stops there.
