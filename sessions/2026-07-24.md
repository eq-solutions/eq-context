## Notes
- **Corrects an assumption an earlier session logged today.** The Coupa PO import session above (PR #996) hit the same class of collision and concluded it was "purely cosmetic, not worth a renumber" — true only when the colliding files haven't been applied yet. Here, all four colliding files were *already* applied under their old names, which changes the mechanics: because the tracking table keys by exact filename, renaming a file that's already been applied orphans its old tracking row and makes the new filename look unapplied again — triggering a real re-apply on the next dispatch, not a no-op rename.
- **That re-apply is usually harmless (recreating the same function definition) — but not always, and this session's own first assessment missed the exception.** One of the two renamed files (`0206`, née `0203_coupa_match_preview_mode.sql`) used a `DROP FUNCTION IF EXISTS (old signature)` + plain `CREATE FUNCTION (new signature)` pair, which is only safe to run once — replayed after its own end state is already live, the bare `CREATE FUNCTION` collides with the function that already exists (`42723` error). This assessment error had a real (if brief and non-customer-facing) consequence: a genuine apply-to-all-tenants dispatch failed on both company databases about 10 seconds after this PR merged — landing in the same batch as a completely unrelated, separately-authored new migration that happened to touch the same function. Nothing was actually broken live (the failed `DROP` target didn't exist, so nothing was ever dropped), but the fleet-wide migration pipeline was blocked until a different concurrent session diagnosed it and shipped a fix (making `0206` use `CREATE OR REPLACE` instead) about two hours later. This session's own final apply dispatch landed just after that fix and confirmed clean. **Lesson: a bare `CREATE FUNCTION` (no `OR REPLACE`) inside a migration is never safe to assume idempotent, even when everything else about the file looks like routine DDL — check for that specific pattern before calling a renumbered-and-already-applied migration "safe to replay."**
