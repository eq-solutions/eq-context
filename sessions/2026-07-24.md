## Notes
- **Squash-merge branch poisoning hit twice this session.** A PR merges (squash) and its branch keeps existing; pushing more commits to that branch does NOT reopen the PR — `gh pr view <old-number>` silently keeps returning stale cached data (`headRefOid` frozen at the pre-merge SHA) instead of erroring, which is what made this hard to notice. Confirmed via `gh api repos/.../pulls/<n> --jq '.state'` returning `"closed"`. Fix both times: `git rebase --onto origin/main <last-pre-squash-commit> HEAD`, `git branch -f <branch> HEAD && git checkout <branch>` (rebase leaves detached HEAD), force-push, open a fresh PR. **Lesson for next time: after any squash-merge, immediately open a new PR (or stop pushing to that branch) rather than assuming the old PR number still tracks it.**
- **Migration numbering isn't safe to assume stays static across a long-running branch.** Authored migration `0198` against main as it stood hours earlier; by the time of the rebase, main had both moved past `0198` for unrelated work AND a separate housekeeping commit had renumbered another in-flight migration into the `0197-0200` range. Renamed to `0201` (next free number) as part of the rebase — but the earlier production dispatch had already applied the migration under the original `0198` filename, so the ledger now needs a reconcile (see Deferred).
- **Near-miss, no actual loss:** re-running an eslint baseline comparison via `git stash` / `git stash pop` — routine technique used successfully several times earlier in the same session — this one time popped a stranger's unrelated stashed WIP (`chore/remove-dead-vendored-platform-apps: tenant-drift-resilience-wip`) because `git stash` found nothing of mine to stash (clean tree) and the chained `pop` fired anyway. One file (`scripts/check-tenant-drift.mjs`) showed a merge conflict from the stray content; restored to HEAD immediately, confirmed the stash entry itself was untouched in the stack (git doesn't drop a stash on conflict) so the original owner's work is intact. **Lesson: don't chain `git stash && ... && git stash pop` unconditionally — check `git stash` actually created a new entry before popping, since stashes are shared across the whole repo/all worktrees, not scoped to the current session.**
- **Real Coupa export shape, confirmed against a live file Royce provided:** columns are `PO Number, Order Date, Status, Acknowledged At, Items, Unanswered Comments, Total` — no site-code or quote-number column ever existed. Both are embedded inconsistently in the free-text Items description (`"SY7- ... QUOTE# SKS-17250"`, `"SY5_Materials for ... Quote#SKS-17257"`, sometimes neither at all e.g. `"CUFT"`). Coupa's `Total` is ex-GST; `quote.total_cents` is GST-inclusive — verified exactly 1.1× across 8 live cross-referenced quotes before relying on it for the value-mismatch check.
- **`netlify/functions/` root filenames must match `[A-Za-z0-9_-]` only** (existing repo CI guard, not new) — Netlify treats every top-level file there as a deployable function, so a `.test.ts` suffix's extra dot breaks it. Test files belong under `_shared/` (or another underscore-prefixed dir). Hit this once this session despite it being a documented, previously-hit trap (per `eq-context` memory of the same class of failure in an earlier eq-shell session) — worth checking new Netlify-function test file placement before first push, not after CI fails.
- **Production migration dispatch (`gh workflow run`) and even a read-only `gh run list` for that specific workflow are blocked by the Claude Code auto-mode permission classifier**, even with explicit in-chat go-ahead from Royce. Correct handling: stop, explain what's needed and why, hand Royce the direct Actions URL to dispatch himself. Do not attempt to route around it via another tool.
