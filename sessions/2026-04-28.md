### 2. Surfaced bug — `updated_at` did not refresh

The verification also showed all three rows still carried `updated_at =
2026-04-12 10:03:35` — exactly 16 days old, identical for all three.
Content was correct, but the freshness signal the brief had just defined
as "done" was structurally broken.

**Root cause traced:** The GitHub Action's PostgREST upsert sends only
`{slug, filename, content}` with `Prefer: resolution=merge-duplicates`.
PostgREST's merge-duplicates only updates columns present in the payload,
so `updated_at` was never touched on conflict. The column's
`DEFAULT now()` only fires on INSERT — meaning every row's `updated_at`
was frozen at its first-ever write.

**Implication:** This retroactively reframes the 2026-04-27 "17 of 30
files stale by 15+ days" audit. Many of those rows had been edited
recently — their frontmatter `last_updated` proved it — but the substrate
audit query was reading a stuck `updated_at` field. The discipline lapse
was real but the metric overstated it. (The "Update Discipline Lapsed"
lesson now carries this retroactive note.)

### 3. Fix — DB trigger + workflow defence-in-depth (commit `fedfaba`)

Royce chose option 3 (both). Two-layer fix:

- **Database trigger** (load-bearing): Migration
  `set_context_files_updated_at_on_update` applied to project
  `urjhmkhbgaxrofurpbgc`. Function `public.set_context_files_updated_at()`
  + trigger `context_files_set_updated_at` `BEFORE UPDATE FOR EACH ROW
  EXECUTE FUNCTION ...`. Stamps `updated_at = NOW()` on every UPDATE
  regardless of payload shape.
- **Workflow defence-in-depth**: `.github/workflows/sync-context.yml` now

- **Backfill**: `UPDATE context_files SET content = content WHERE slug IN
