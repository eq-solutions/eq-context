# Session 2026-07-01 — Cert-import 500 root-caused + fixed (async payload wall)

## Built
- PR #563 merged (b729eed) — calibration cert import 500 fixed. Browser now uploads PDFs via synchronous upload-asset-cert, hands the background parser JSON URLs only; parser fetches bytes server-side (SSRF-guarded to SUPABASE_URL origin). Parse-time URLs reused at commit (one upload per PDF).
- Stale-branch prune deferred items also closed earlier today (see part j/part-b history): eq-shell 49->9 local, 14->5 remote.

## Decided
- Royce authorized merge of #563 (auto-deploys to core.eq.solutions).

## Deferred
- Remove dead cert-import-parse.ts (old sync parser, now unused; kept per no-delete rule).
- Verify cert import live once deploy is green (hard-refresh for new panel JS).

## Notes
- **ROOT CAUSE (load-bearing):** a Netlify `-background` function is invoked ASYNCHRONOUSLY; async Lambda event-payload limit ~256KB vs 6MB synchronous. POSTing multi-MB multipart bodies to a background fn = platform rejects the invocation BEFORE the handler runs -> generic "Internal Error. ID: <ulid>" (a Netlify request id, NOT Sentry) with ZERO function logs. Never route file bytes through a background function — upload via a sync endpoint, pass URLs.
- Diagnostic that nailed it: `netlify logs --source functions --function <name> --since 24h` showed background fn had 0 handler logs while the sync upload-asset-cert logged fine. "No handler logs" = rejected at platform layer.
- `tsc -b` covers netlify/functions via tsconfig.netlify.json (referenced by root tsconfig).
