# Session 2026-07-12 — EQ Field: sbFetchAll `order=id` "bug" verified nspb-only, no change

## Built
- Nothing. Investigation + live-DB verification only; no code changed, no PR, no deploy.

## Decided (Royce)
- **No change** to EQ Field's three `sbFetchAll` call sites (index.html:8488 `team_members`, timesheets.js:248 `timesheet_locks`, supabase.js:1765 `project_targets`). The proposed port of the sks-nsw-labour `order=id` 400 fix (nspb v3.10.92 / PR #59) does not apply here.

## Notes
- **Premise was wrong for EQ Field.** `sbFetchAll(path, orderBy='id')` 400s only when the target relation has no `id` column. That's true on the standalone **nspb** DB (`team_members` PK=`team_id,person_id`, `timesheet_locks` PK=`week_key,org_id`, no `id`) — a separate repo/DB EQ Field never touches. On EQ Field's planes every relation these calls hit **has `id`**, so `order=id` returns 2xx.
- **SKS (ehow):** JWT tables rewrite to `app_data.field_*` twin views. `field_team_members` (exposes `id` + renames `staff_id`→`person_id`, which is why the app maps `r.person_id`), `field_timesheet_locks`, `field_schedule`, `field_timesheets` — all expose `id`. Base `app_data.team_members`/`timesheet_locks` also carry an `id` PK.
- **eq (zaap):** verified by executing the compiled query (`… ORDER BY id LIMIT 1000 OFFSET 0`) against both possible resolutions — `app_data.field_timesheet_locks`/`public.timesheet_locks` and `app_data.field_project_targets`/`public.project_targets` — all executed clean (no 42703 → 2xx; 0 rows = empty sandbox, still 2xx). `team_members` is `ORG_SLUG==='sks'`-gated so it never runs on eq. `project_targets` loader is Enterprise-tier-gated.
- **Full-repo sweep:** every other default-`order=id` `sbFetchAll` call (`schedule`, `timesheets`, `nominations`, `pending_schedule`, `tenders`) also resolves to a relation with `id`. No id-less `order=id` 400 exists anywhere in EQ Field.
- **Do NOT port the nspb fix here** — swapping the unique `id` sort for a non-unique key (e.g. `week_key`) would, if anything, mildly hurt the offset paginator's determinism; it's a no-op at best. Saved to memory as `project_sbfetchall_orderid_safe`.
- pending.md needs no edit — line 454 already ticks the `sbFetchAll()` pagination work (2026-07-10); nothing here is deferred.
