**Audit findings worth knowing:**

- The DB is more tenant-shaped than the roadmap assumed. `org_id uuid` already
  exists on every core table; `organisations` is the tenancy root. We do **not**
  need a new `tenants` table.
- RLS is enabled on only 3 tables (`leave_balances`, `checkins`,
  `ts_reminders_sent`) and most policies are permissive (`USING (true)`). Real
  tenant isolation lives at the app layer today, not the DB.
- Auth is custom HMAC tokens (not Supabase Auth). PIN compare is plaintext per
  the established memory rule.
- PostHog is initialised but feature flags are not used anywhere in code yet.

### Decisions locked

1. **Sprint scope:** Phase 1 only this sprint (PostHog flag wrapper +
   project-hours feature + 5-tier role system). Phase 2 (multi-tenancy
   foundation) deferred. **Trigger to start Phase 2:** first self-serve trial
   signup is on a calendar OR ~3 customers manually provisioned and the
   operational cost is biting.
2. **Tenancy boundary:** Multi-tenancy lives inside `ktmjmdzqrogauaevbktn`
   (eq-solves-field) only. SKS Labour stays on `nspbmirochztcjijmcrx`,
   untouched. Future enterprise customers handled case-by-case.
3. **Auth surface:** `verify-pin.js` will mint a Supabase-native JWT signed
   with the project's JWT secret, with `app_metadata.tenant_id` and
   `app_metadata.eq_role`. RLS uses the standard
   `((auth.jwt() -> 'app_metadata') ->> 'tenant_id')::uuid` pattern. PIN UX
   preserved.
4. **Tenant URL convention:** `eq.solutions/field/<slug>/`. Path-based slug
   resolution, single Netlify site, no per-tenant subdomains. Routing
   infrastructure (Cloudflare Worker proxy or `field.eq.solutions` subdomain)
   deferred until Phase 2 trigger fires.
5. **PIN model:** Single PIN per tenant, stored on
   `organisations.tenant_pin`. Role is derived from `people.role`, not from
   which PIN matched. The per-role PIN env-var pattern is dropped — doesn't
   scale past 2 tenants.
