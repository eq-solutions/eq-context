# Session 2026-07-04 — ehow security review: 6 real cross-tenant/anon bugs found and fixed live, WARN count 206 → 145

*Requested scoped review of `ehow` (ehowgjardagevnrluult, live DB behind EQ Service + EQ Field for SKS) after Supabase advisor flagged 206 WARN-level security findings. Read every `SECURITY DEFINER` function individually rather than trusting the advisor's category counts — that discipline is what surfaced the real bugs.*

## Built
- Closed 4 confirmed anon-exposure gaps: `service.list_active_supervisors()` (leaked every tenant's supervisor emails/names to unauthenticated callers), `service.pm_calendar_for_supervisor(...)` (IDOR — checked target's role, never caller's identity), `public.get_defect_counts(p_tenant_id)` and `public.eq_mark_expired_quotes()` (both anon-callable with no auth). Root cause: Postgres's default PUBLIC-grant-on-create was never revoked. Fix was grant-only (no function-body changes) after verifying every real caller uses the service-role key server-side.
- Hardened `search_path` on 9 functions with no pinned path.
- Revoked inert anon/authenticated grants on 22 trigger-only functions (advisor-noise cleanup, zero functional risk — Postgres blocks direct invocation of `RETURNS trigger` functions regardless of grant).
- Relocated the unused `vector` extension out of `public` (confirmed zero columns use it first); left `pg_net` alone (not relocatable without drop/recreate risk).
- After Royce pushed back on "145 remaining is fine," read all 83 `authenticated`-only `SECURITY DEFINER` functions individually and found 2 more real bugs of the same caller-controlled-tenant shape: `eq_get_job_creation` (2-arg overload) and `get_effective_notification_prefs` — both fixed by restricting to `service_role` after verifying every real caller already uses it.
- Built a pricing-config role gate (10 functions: `eq_upsert_pricing_config`, `eq_replace_pricing_bands`, `eq_upsert_pricing_material`/`_product`, `eq_upsert_quote_template`, `eq_upsert_rate_preset`, and their archive variants) — added a shared helper `eq__assert_pricing_role()` that checks `service.tenant_members.role IN ('manager','supervisor')` live, never trusting a JWT role claim.
- Verified (not assumed) that `field_person_by_user_id` is now fully closed: eq-field PR #387 (built by Royce's spawned session `task_2f75aab3`) moved the Shell→Field person lookup server-side into `verify-pin.js` using a new `EHOW_SERVICE_ROLE_KEY`; confirmed the code, the env var, and the production deploy all independently before checking the DB grant — found the grant already locked to `service_role` only, nothing left to run.

## Decided
- Royce: fix all 4 initially-found anon-exposure gaps immediately (not partial/deferred).
- Royce: safety-record self-approval (`approve_safety_record`/`submit_safety_record`) — leave as-is; field-crew reality (sole-charge techs) makes a distinct-approver requirement impractical.
- Royce: pricing-config writes restricted to manager + supervisor roles (not manager-only).
- Royce: reversed an in-flight recommendation — initially proposed building a new eq-roles→JWT Auth Hook bridge for role checks; reversed after finding `eq-solves-service/supabase/migrations/0134_role_crosscheck.sql` showed EQ Service had already built and abandoned exactly that pattern (JWT role claims went stale/drifted from real membership). Used the already-proven `tenant_members.role` live-join pattern instead.
