# Session 2026-06-28 (part b) — Shell↔Service branding audit + token refresh + admin hub tiles

## Built

- **`ShellTokenRefresh` component** (`components/ui/ShellTokenRefresh.tsx`) — client component that sends `REQUEST_SHELL_TOKEN` to Shell parent 5 min before the 4h `eq_service_jwt` cookie expires, handles `SHELL_TOKEN_RESPONSE`, re-exchanges via `/api/shell-auth` to mint a fresh cookie, then reschedules. No-ops when not in a Shell iframe. eq-service PR #364 merged.
- **Admin link in Shell iframe inline nav** (`app/(app)/layout.tsx`) — added `Admin` link to the `isShellIframe` top nav bar for `isAdmin` users (manager role). Previously no way to reach `/admin` from within the Shell iframe (Service sidebar is fully hidden when `isShellIframe=true`). PR #364 merged.
- **EQ Service section in Shell Admin hub** (`src/pages/AdminHub.tsx`, eq-shell) — 8 tiles (Report settings, Media library, Archive, Imports, Backup, Activity feed, Today, Connected apps) linking to `/<tenant>/service/admin/*`. Gated on `moduleEnabled(session, 'service')`. eq-shell PR #518 merged.

## Decided

- **Generic settings belong in Shell, app-specific settings in Service.** Shell Admin hub is the home for cross-suite admin (users, roles, security groups, workspace settings). Service-specific admin pages (report templates, media, archive, imports, backup) stay in Service — not rebuilt in Shell.
- **Service sidebar is hidden when embedded** (`{!isShellIframe && <Sidebar />}`) — confirmed from code. Admin access for Shell iframe users must come from Shell's nav, not Service's sidebar. Steelman of "should this be in Service?" — answered: yes, but it needs to be reachable via Shell.
- **Branding wiring is correct.** Shell mints `brand_color` + `brand_logo_url` into the JWT (`token-exchange.ts` for `aud=service` only). `shell-auth` re-mints them into the 4h cookie and write-behind syncs to `tenant_settings`. `getTenantSettings` prefers JWT claims (always current). No action needed on branding itself.
