# Session — 2026-06-13

## What was done

### EQ Service iframe loading fix — eq-shell PR #334

**Problem:** EQ Service showed "Loading EQ Service…" indefinitely when accessed from Shell (`core.eq.solutions/sks/service`). Broken for 1+ weeks. Royce: "isn't working — can we fix this once and for all."

**Investigation:**
- Read all TOKEN MODE auth code paths: proxy.ts, shell-auth/route.ts, (app)/layout.tsx, shell/page.tsx, ShellReadySignal.tsx
- Verified cookies (SameSite=Lax, same-site for eTLD+1 eq.solutions), proxy TOKEN MODE fast path, shell-auth JWT fast path, MFA bypass for shell sessions, ShellReadySignal target origin
- Checked Sentry (eq-solves-service): 0 related issues
- Verified no frame-ancestors restriction in Service netlify.toml
- Confirmed `app/page.tsx` redirect is client-side (no second iframe onLoad)

**Root cause:** `ServiceIframe.tsx` `onIframeLoad` fallback timer was `12_000ms` with a comment "cover the OTP round-trip". `COOKIE_AUTH = false` (TOKEN MODE) was made permanent, eliminating the OTP round-trip entirely. TOKEN MODE handshake completes in ~2-3 seconds. Users who refreshed before 12s restarted the clock — indefinitely showing the loading screen.

**Fix:** `12_000` → `4_000` ms fallback. Added Sentry breadcrumbs:
- `EQ_SERVICE_READY` received (success path, `level: info`)
- Fallback fired — EQ_SERVICE_READY not received in time (`level: warning`)

**File changed:** `src/pages/ServiceIframe.tsx` (`onIframeLoad`, line ~175)

**PR:** [#334](https://github.com/eq-solutions/eq-shell/pull/334) — merged 2026-06-13T00:44:57Z. Netlify deploy confirmed live (deploy `6a2ca798cc9a6c000854ab11`, published 2026-06-13T00:45:48Z).
